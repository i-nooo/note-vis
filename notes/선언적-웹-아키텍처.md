> 출처: [웹 브라우저 엔지니어링](https://browser.engineering/intro.html)

<br/>

- 웹은 하이퍼링크로 연결된 정보의 네트워크이다.
- 사용자는 브라우저 라고 하는 사용자 에이전트를 사용하여 웹을 탐색한다.
- 정보는 HTTP(HyperText Transfer Protocol)를 통해 요청되고 HTML 문서 형식으로 구조화된다.
- 문서는 내용이 아닌 URL(Uniform Resource Locator)로 식별되며 동적으로 생성될 수 있다.
- 웹 페이지는 이미지, 비디오, CSS(Cascading Style Sheets), JavaScript를 포함한 다양한 형식의 보조 자산에 링크할 수 있다.
- 이러한 모든 구성 요소는 개방적이고 표준화되어 있으며 무료로 사용하거나 재사용할 수 있다.

<br/>

웹 브라우저의 동작을 가능하게 하는 핵심은 제어의 역전(inversion of control), 제약 프로그래밍(constraint programming), 그리고 선언적 프로그래밍(declarative programming) 의 구현에 있다.

> [!NOTE]
> **제어의 역전**
>
> 일반적으로 전통적 프로그래밍은 개발자가 “순서대로” 동작을 직접 제어하지만, 하지만 웹에서는 브라우저가 중심이 되어 렌더링, 이벤트 처리, 동기화 등을 맡고, 개발자는 선언적으로 지시만 한다. 웹 브라우저는 거대한 프레임워크라고 볼 수 있다.
> \
> **제약 프로그래밍**
>
> 웹 레이아웃은 절대 좌표 기반이 아니라, “이 요소는 저 요소 옆에 있어야 한다”, “이 너비는 부모의 50%여야 한다” 같은 제약식으로 표현된다. (flexbox, grid)
> \
> **선언적 프로그래밍**
>
> DOM 변경, CSS 속성을 바꾸는 작업은 "지금 당장 화면을 다시 그려라"라는 명령이 아니라, "이 상태가 바뀌었다"는 사실을 브라우저에 알리는 것(상태변경 선언)이다. 브라우저는 이 변경을 즉시 처리하지 않고, 내부 큐에 기록만 해둔다. 실제 반영 시점은 브라우저의 최적화 루틴에 따라 지연될 수 있다. 선언적 스타일 덕분에 개발자의 관점에서는 변화가 “즉시 적용되는 것처럼” 보이지만, 내부적으로 브라우저는 게으른(lazy)[^1] 방식으로 동작하여, 후속 API 호출이나 실제 화면 표시가 필요할 때까지 적용을 지연시킬 수 있다. 이는 성능 최적화(레이아웃 병합, repaint 지연 등)를 가능하게 한다.

<br/>

**Q**. 왜 게으른 방식을 사용할까? (= 왜 미룰까?)
**A**. 계산 비용 때문이다. 레이아웃 계산, 스타일 재계산, 페인팅, 합성은 모두 고비용 작업이며, 중간에 여러 변경이 연속으로 발생하면 각 변경마다 즉시 처리하는 것은 최악의 성능 선택이 된다. 브라우저는 이를 피하기 위해 “아직 결과가 필요하지 않다”고 판단되면 계산을 보류한다.

<br/>

즉, 웹 브라우저를 단순히 “표시 도구”로 보지 않고, 제약 해결기(constraint solver) + 이벤트 루프 + 선언적 실행 환경이라는 큰 소프트웨어 아키텍처로 이해해야 한다.

---

**Q**. 이 세 가지(제어 역전, 제약 프로그래밍, 선언적 프로그래밍)가 프론트엔드 프레임워크(React, Vue, Svelte 등)에 어떻게 계승·확장되었는지
**A**. 세 가지 개념은 프론트엔드 프레임워크에서 그대로 재현된 것이 아니라, 브라우저 위에 한 겹 더 추상화되어 계승되었다. 핵심은 “브라우저가 하던 일을 프레임워크가 일부 선점하고, 브라우저의 게으름을 더 조직적으로 활용한다”는 점이다.

- 먼저 제어의 역전부터 보자. 브라우저에서의 제어 역전은 “이벤트 루프·렌더링·동기화의 주도권이 브라우저에 있다”는 구조였다. 프레임워크는 여기서 한 단계 더 나아가 컴포넌트 생명주기와 상태 전파의 제어권까지 프레임워크가 가져간다. 개발자는 DOM API를 직접 호출하지 않고, 상태(state)를 바꾸는 행위만 수행한다. 그러면 프레임워크가 “언제 렌더링할지”, “어떤 컴포넌트를 다시 계산할지”를 결정한다.
- 브라우저의 제약은 레이아웃 중심이었다. “이 요소는 저 요소의 자식”, “이 크기는 비율로 결정” 같은 공간 제약이다. 프레임워크는 이를 상태 간의 '논리적 제약'으로 확장했다. 예를 들어 “이 값이 바뀌면 저 값도 자동으로 바뀌어야 한다”, “이 상태가 true면 이 UI는 반드시 존재해야 한다” 같은 관계를 코드로 명시한다. 이는 수식 기반 제약이 아니라 의존성 그래프 기반 제약이다. 프레임워크는 "무엇이 무엇에 의존하는지"를 추적하고, 필요한 것만 다시 계산한다.
- 선언적 프로그래밍은 가장 직접적으로 계승되었지만, 동시에 가장 크게 확장되었다. 브라우저에서 선언성은 “DOM과 CSS를 상태로 기술한다”, "이 DOM은 이런 구조다"는 수준이었다. 프레임워크에서는 선언 대상이 UI 트리 전체로 확대된다. 개발자는 “이 상태일 때 UI는 이런 구조여야 한다”를 선언하고, 프레임워크는 이전 UI를 기억하고, 차이를 계산하고, 실제 DOM을 고치는 일을 수행한다. 여기서 중요한 점은, 이 선언성이 게으름을 전제로 설계되었다는 것이다. 프레임워크는 상태 변경이 발생해도 즉시 DOM을 건드리지 않는다. 내부적으로 업데이트를 큐에 쌓고, 브라우저의 렌더링 타이밍에 맞춰 병합한다.
- 프레임워크는 “마법처럼 편한 도구”가 아니라 브라우저의 추상화 비용을 대신 관리해주는 시스템이다. 프레임워크를 쓰면서 성능 문제가 발생하는 이유는, 이 제약·게으름·제어 역전의 경계를 무의식적으로 침범했기 때문이다(예: 불필요한 상태 분할, 과도한 파생 상태, 렌더링 강제).
